01
Choice — BYO vs managed
 The fork in the road. Most tenants don't know there's a choice; this screen makes the trade-off visible (deliverability + brand) and lets the ones who can't connect a domain skip without dropping out of onboarding entirely. Picks one of two cards: "Use my own domain" (BYO) or "Use a managed domain". Nothing yet — just a routing decision. We log the choice for analytics (drop-off where). No Entri call yet. If they pick managed → skip the next 4 screens entirely, mark Engage with a useManagedDomain flag, advance to Email step. If BYO → go to screen 02.
02
Root domain
 We need the apex domain (acmeinsurance.com) before we can build any DNS records. Pre-filled from the business website to remove typing and reduce typos that would silently break verification. Confirms or edits the prefilled root domain. Root domain string. No Entri call yet. Stored on our side. We do basic format validation (is it a real-looking FQDN?) but no DNS lookup.
03
Subdomain
 Sets the subdomain prefix (send) so the full sending domain becomes send.acmeinsurance.com. Postmark/SES need the full subdomain to mint DNS records. Per product policy, we never let tenants send from the apex domain — we don't trust untrained customers not to damage email reputation on a domain they also use for personal mail. Enters subdomain prefix (default: send). Sees a live preview chip showing the full domain. Clicks Continue. Subdomain string. We then call entri.checkDomain on Continue (gating). First Entri call: entri.checkDomain(fullDomain, config) — fires when the user clicks Continue, before advancing. Returns { provider, setupType: "Automatic" | "Manual", NSSupport }. We use it to gate: if the provider is supported we advance; if unsupported we still advance but route the user toward "Set up manually" on the next screen and surface a heads-up. This is a paid add-on not included in the base startup plan — Entri is providing one-off pricing.
04
DMARC opt-in
 DMARC isn't strictly required by Postmark, but Gmail/Yahoo bulk-sender rules effectively require it for any tenant sending >5k/day. Splitting it as its own screen with a default-on toggle gives tenants visibility (so they don't get a surprise DNS record they didn't approve) and a way to opt out if they already manage DMARC themselves. Now placed before the Entri/manual screen so the full DNS payload is committed before we launch the modal — Entri runs once with all records (per Felipe at Entri). Confirms DMARC is on (default) or toggles off. Boolean: include DMARC record? No Entri call yet. The choice gets stashed locally; we'll append the DMARC record to the dnsRecords[] array on the next screen if the toggle is on. Format: _dmarc.send TXT "v=DMARC1; p=none; rua=mailto:dmarc@...". Because we're always writing to a fresh subdomain, we don't need Entri's "Validate DMARC" feature (would only matter if we were ever overwriting an existing DMARC record on the apex).
05
Connect with Entri / Set up manually
 The "are you ready to give us write access to your DNS?" moment. Gives the tenant a beat to opt out, see the legal terms, and choose manual fallback if they don't trust auto-setup or their provider wasn't detected. The submit button on this screen is what actually launches the Entri modal. Picks "Connect with Entri" (most users) or "Set up manually" (power users / unsupported providers / IT pushback). Choice of automatic vs manual path. Backend + Entri both kick off here. We call our POST /api/sending-domain endpoint, which: (1) calls Postmark POST /domains to register the sending domain and gets back the DKIM TXT + Return-Path CNAME records, (2) translates them into Entri's dnsRecords[] shape, (3) appends the DMARC record from screen 04 if opted in, (4) mints a 60-min Entri JWT via POST https://api.goentri.com/token, (5) returns the JWT + records to the browser. The browser then calls entri.showEntri({ applicationId, token, userId, prefilledDomain, dnsRecords }) to open the modal with the full payload at once.
05.5
Entri modal
Not our UI 		The actual DNS-write moment. Entri owns this UI; we just hand it the JWT and records, and listen for callbacks. Inside the modal: tenant logs into their DNS provider via OAuth (or copies records manually), and Entri writes the records on their behalf. Logs into DNS provider (Cloudflare, GoDaddy, etc.) inside the Entri modal. Or copies records manually. Or hands off to IT via Entri's "forward login" feature. DNS records get written. Not our screen — Entri's iframe modal. We launch with entri.showEntri({ applicationId, token, userId: "tenant_<id>", prefilledDomain, dnsRecords }). We listen for onSuccess (records written client-side) and onEntriClose (modal closed, returns { success, setupType, provider, lastStatus }). We persist setupType and provider for ops debugging.
06
Verifying
 After the Entri modal closes successfully, DNS records have been written but propagation isn't instant (1–5 min typical). This screen shows per-record progress and crucially says "you can leave this page" so users don't bounce while watching a spinner. Sees per-record status (DKIM, Return-Path, DMARC). Can re-check now, or click "continue setup" and verify in background. Per-record propagation status, driven by Entri webhooks. No more outbound Entri calls — we listen for webhooks. Per Felipe at Entri: "at that point you're going to only listen to the webhook and the record propagation." We need a webhook handler at e.g. POST /api/webhooks/entri that receives domain.added with propagation_status: "pending" | "success" | "failed" for each record, plus domain.flow.completed. Webhook signing uses Entri-Signature-V2 + Entri-Timestamp headers (SHA-256 over webhook_id + timestamp + client_secret). All production webhook traffic comes from a fixed Entri IP — IP-allowlist at the edge. Once all expected records report success, we flip the domain to Verified and the tenant can send.
07
Settings — list view
 The post-onboarding home for sending domains. Tenants need to (a) see what's connected, (b) add a second domain (transactional + marketing pattern), (c) re-verify or remove existing ones, (d) pick a default. Always shown alongside the managed-domain row as a fallback that's never deletable. Views status per domain. Adds a new domain (re-enters the flow at screen 01). Re-verifies, sets default, removes. Existing domains, status, last-verified timestamp, default flag. Re-enters the same flow when "Add domain" is clicked. Re-verify is a user-initiated check (we don't have streaming webhooks for an idle domain) — backend does a fresh DNS lookup against Postmark's verify endpoints and updates our record. Remove calls Postmark DELETE /domains/{id} and deletes our row.
07b
Settings — empty state
 A tenant who skipped BYO during onboarding. The framing matters: "you have no domains" is technically true but doesn't communicate the real state ("you're sending from a managed domain, here's what you're missing"). Reads the framing, decides to upgrade. Nothing — just a CTA. Re-enters the flow at screen 01 (or skips straight to 02 since the choice is implicit).
08
Disconnected — banner + table state
Post-MVP 		A previously verified domain stopped resolving — IT changed providers, someone deleted the record, an SPF record got overwritten by another tool, etc. Per-domain because in multi-domain setups one can fail while others stay healthy. Deferred from MVP. Detection requires Entri Monitor ($50/mo for up to 600 domains on the startup plan). The Entri team flagged the most likely real-world failure as customers using a second email tool that overwrites SPF instead of merging — worth revisiting once we see actual disconnect rates in production. Sees the banner, clicks "Reconnect" (re-launches Entri with the same records) or "Forward fix to IT". The specific record that's missing (DKIM / Return-Path / DMARC) and what the value should be. When we add this: subscribe to Entri Monitor and listen for domain.record_missing and domain.record_restored webhooks. Reconnect re-launches entri.showEntri() with the original record set; no need to re-mint or re-create the Postmark domain. Until then, tenants who silently disconnect will surface only via deliverability complaints.

Flow rules worth calling out

  • Steps 02–05 collapse into one onboarding wizard step in the existing left sidebar (Brand → Sending domain → Email → Business → SMS → Merge fields). The 4-of-4 sub-stepper at the top of the right pane handles internal progression.
  • Entri runs once, with the full DNS payload at once. This is why DMARC opt-in (04) comes before the Entri/manual screen (05): we want DKIM + Return-Path + DMARC all decided before we open the modal so the user goes through the OAuth dance only one time. Confirmed by Felipe at Entri.
  • The Entri modal (05.5) is the only non-platform UI in the flow. Everything else is ours to design and own.
  • Webhooks are the verification mechanism, not polling. After entri.showEntri resolves, we don't poll — we listen. Entri pushes domain.added events with per-record propagation status, plus domain.flow.completed. Build the webhook handler with HMAC verification (Entri-Signature-V2 over webhook_id + timestamp + client_secret, SHA-256, hex), idempotency on webhook_id, and an IP allowlist for Entri's fixed origin IP. This is the load-bearing integration boundary on our side.
  • checkDomain is a paid one-off add-on. Not on the base startup plan; Entri is providing per-feature pricing. Treat it as a Continue-time gate from the subdomain step (screen 03), not an inline blur hint. The result decides whether the next screen leans toward "Connect with Entri" or "Set up manually".
  • Apex-domain sending is disallowed. Product policy: tenants can only send from a subdomain we provision (send.acme.com, never acme.com). Untrained customers shouldn't be able to torch the reputation of the domain that also hosts their personal email.
  • Managed domain is always present in the background, even for BYO tenants. The managed row in screen 07 is informational and not removable. Future state: if a BYO domain disconnects, we automatically fall back to the managed one so sends keep going.
  • SPF, Entri Monitor, and Validate-DMARC are out of scope for MVP. SPF: Postmark doesn't require it for DKIM-aligned auth, and we're writing to a fresh subdomain. Entri Monitor (screen 08): $50/mo for 600 domains on the startup plan, deferred until we see real-world disconnect rates. Validate-DMARC and Advanced DMARC options: only matter if we ever overwrite an existing DMARC on the apex, which we don't.

How this state is driven (per Felipe at Entri): after entri.showEntri resolves we don't poll Postmark — we listen.
