Engage Email Channel & BYO Sending Domain — implementation reference

This doc is the engineering reference for the Engage onboarding redesign and the bring-your-own sending domain feature (Linear ticket ENG-2161, sub-tasks ENG-2670–ENG-2749). The clickable prototypes are the source of visual truth — this doc captures the flow tables, integration contracts, and design rationale. Start with the prototypes, come here for the integration details and the "why."

01

Overview · what's shipping

The 60-second version

Problem. Tenants currently send Engage email from a platform-controlled sending domain. Recipients see "via mail.platform.com" in Gmail/Outlook headers, which hurts deliverability and conflicts with Google/Yahoo bulk sender rules (>5k/day to Gmail). Partner-side blocker for tenant adoption of Engage.

Solution. Every Engage tenant gets a platform-managed sending domain provisioned automatically. Each tenant can optionally connect their own branded domain (e.g. send.acmeinsurance.com) via Entri for better deliverability. Multi-domain supported — tenants can split marketing and transactional onto different subdomains. Per-domain sender identity (prefix / display name / reply-to) so each domain has its own "from" line.

Two surfaces. Onboarding is a 6-step wizard (Brand → Business Info → Email Channel → SMS → Merge Fields → Automations). Settings is a permanent sidebar with the same five non-completion tabs. The Email Channel step + tab is where managed and BYO live side-by-side. The BYO sub-journey (Root → Subdomain → DMARC → Connect → Verifying) runs as a sidebar-swap sub-flow in both places.

Brand Settings
 Captures the business name and website. The website seeds the BYO root domain on the Email step's sub-journey — never asks the user to retype it. Confirms business name and website. Business Information
Hard gate 		CAN-SPAM (US) and equivalent laws require a physical postal address in marketing email. SMS carriers require business verification before they'll assign a sending number. Both are upstream blockers — must be filled before Engage can send at all. Fills in DBA, address line 1, city, state, ZIP, country. Address line 2 optional. All required fields. Continue button is disabled until DBA + address + city + state + ZIP are populated. Stored in our DB. Used in email footers (CAN-SPAM compliance) and submitted to SMS provider on the SMS step.

Email Channel · managed card
Always set up 		Every tenant gets a managed domain provisioned automatically so they can send mail immediately. The managed card collects the sender identity (prefix, display name, reply-to) and shows an inbox preview. Confirms or edits prefix (e.g. marketing), display name (e.g. "Acme Insurance"), reply-to (e.g. info@acmeinsurance.com). Inbox preview updates live. Prefix, display name, reply-to. Display name is required. Reply-to has an auto-suggest derived from the sending root domain and a warning chip if the user enters a cross-root reply-to (deliverability nudge). Managed domain is provisioned at tenant-create time, before onboarding. ESP (Postmark or SES per partner) already has DKIM/Return-Path live for the managed domain.
03a
Email Channel · BYO card
Optional, recommended 		Better deliverability + brand trust. Visually framed as a parallel path under the managed card so users see they have a choice but don't feel forced. "Connect your domain" is the only action on this card; everything else lives in the sub-journey. Clicks "Connect your domain" if they want BYO. Otherwise skips and continues. No data collected on this card itself. No external calls until they enter the sub-journey.
03b
BYO sub-journey · Root domain
Sub-step 1 of 4 		We need the apex domain (acmeinsurance.com) before we can build any DNS records. Pre-filled from the Brand step's website so the user just confirms. Confirms or edits the prefilled root domain. Root domain string. No Entri call yet. Stored on our side. Basic FQDN format validation only.
03c
BYO sub-journey · Configure sending domain
Sub-step 2 of 4 		Collects everything we need to know about the BYO domain in one screen: the subdomain prefix AND the per-domain sender identity. Mirrors the managed-card form so the user sees they're configuring the same kind of thing, just under their own domain. Critically: identity is per-domain — the BYO domain doesn't inherit silently from managed. Enters subdomain (default send). Edits prefix, display name, reply-to. Sees a live preview chip with the full sending domain and an inbox preview. Subdomain, prefix, display name, reply-to. BYO identity seeds from the managed identity on first entry — user can edit any field. Reply-to auto-suggests info@<byo-root-domain>. First Entri call (gated on plan): entri.checkDomain(fullDomain, config) if we have it enabled. Returns { provider, setupType, NSSupport }. Used informationally on the next screen — not a hard gate.
03d
BYO sub-journey · DMARC
Sub-step 3 of 4 		DMARC isn't strictly required by Postmark but Gmail/Yahoo bulk-sender rules effectively require it for any tenant sending >5k/day. Default-on toggle gives visibility (so users don't get a surprise DNS record) and an opt-out for tenants who manage DMARC externally. Confirms DMARC on (default) or toggles off. Boolean: include DMARC record. We append a DMARC record to dnsRecords[] before launching Entri if the toggle is on. Format: _dmarc.<subdomain> TXT "v=DMARC1; p=none; rua=mailto:dmarc@...". If off, skip it.
03e
BYO sub-journey · Connect
Sub-step 4 of 4 		The "give us write access to your DNS?" moment. Gives the tenant a beat to opt out, see legal terms, or pick the manual fallback for unsupported providers. Picks "Connect with Entri" (most users) or "Set up manually." Choice of automatic vs manual. Backend kicks off here. POST /api/sending-domain: (1) calls Postmark POST /domains to register the sending domain, gets back DKIM TXT + Return-Path CNAME; (2) translates into Entri's dnsRecords[]; (3) appends DMARC record if toggled on; (4) mints a 60-min Entri JWT via POST https://api.goentri.com/token; (5) returns JWT + records to the browser. Browser launches entri.showEntri({ applicationId, token, userId: "tenant_<id>", prefilledDomain, dnsRecords }).
03f
Entri modal
Not our UI 		The actual DNS-write moment. Entri owns this UI. Logs into DNS provider via OAuth inside the Entri modal. Or copies records manually. Or hands off to IT via Entri's "forward login" feature. DNS records get written by Entri. We listen for onSuccess and onEntriClose on the browser side. Server-side we listen for Entri webhooks (see §4). We persist setupType and provider from the close callback for ops debugging.
03g
BYO sub-journey · Verifying
Sub-step 5 (auto) 		Entri saying "records are written" ≠ Postmark saying "I can DKIM-sign for this domain." Propagation takes 1–5 minutes typically. Per-record progress UI keeps the user oriented. Watches the animated per-record progress. Can click "Continue while verifying →" to advance to SMS without waiting; verification keeps polling in the background and the user gets notified when it lands. Postmark verification status per record (DKIM, Return-Path, DMARC). No more Entri calls. Backend polls Postmark PUT /domains/{id}/verifyDkim + PUT /domains/{id}/verifyReturnPath on backoff: every 30s for 5 min, then every 2 min for 30 min, etc. Frontend polls our status endpoint. Also receives Entri domain.added webhooks server-side as a fast signal to trigger immediate Postmark verify. When both Postmark calls return verified, domain flips to Verified.
04
SMS Channel
 Submits Business Info to the SMS provider for verification. Provider assigns a sending number once approved (3–7 business days). Clicks "Request approval." Continues. Submission of existing Business Info — no new fields. API call to SMS provider. Out of scope for Entri integration.
05
Merge Fields
 Personalization defaults (First Name, Last Name) with fallback strings for when we don't have the customer data. Sets backup text. Skippable. Merge field definitions. Stored in our DB. No Entri interaction.
06
Automations
 Activates recommended automations (welcome series, birthday, thank-you). Existing OnboardingAutomationList component. Toggles automations on/off. Clicks "Save & go to dashboard →". Per-automation enabled boolean. No Entri interaction.

Onboarding-specific rules

  • The BYO sub-journey is a sidebar swap. When the user clicks "Connect your domain" on the Email step, the left sidebar (Brand · Business · Email · SMS · Merge · Automations) replaces with "← Email Channel" parent button + 4 indented BYO sub-steps. Right pane swaps to the current sub-step's content. Footer Previous/Continue map to sub-journey navigation.
  • Continue while verifying. The Verifying screen exists in the sub-journey, not as a blocking step. Users can click "Continue while verifying →" to advance to SMS while the DNS records finish propagating. Verification continues in the background; the user gets notified (and the DNS section flips to Configured) when it lands.
  • Bailing out preserves partial state. If the user closes the sub-journey before finishing, the BYO sub-step they were on is saved as state.byo.pausedAt. When they return to the Email step, the DNS section shows the paused variant with a Resume button — no data lost.
  • BYO identity seeds from managed. On first entry to the BYO sub-journey, state.byo.{prefix,displayName} are seeded from the managed-domain values (the user already filled these in). replyTo stays blank so the auto-suggest placeholder shows info@<byo-root> instead of carrying over the managed reply-to.
  • Reply-to deliverability nudge. Every reply-to field auto-suggests info@<sending-root-domain> as a placeholder. If the user types a reply-to whose root domain differs from the sending domain's root, show the soft warning chip. Doesn't block — just signals the deliverability risk.

Email Channel · list view
Default tab view 		Single home for all email-sending domains. Each row is a clickable card showing domain, status pill, sender summary (prefix@fullDomain), purpose label, last verified time, and a Default pill. Clicking a row opens that domain's detail page. Scans the table. Clicks a row to configure. Clicks "+ Add domain" to enter the BYO sub-journey for a new domain. Email Channel · domain detail
Master → detail 		Per-domain configuration page. Two sections: Sender identity (prefix, display name, reply-to, inbox preview) and DNS connection (Voice-style status card with five variants). Plus a Danger zone with Remove for non-managed domains. Edits sender identity fields (per-domain save bar appears on dirty). Manages DNS via the DNS section. Removes the domain via Danger zone. Per-domain prefix, display name, reply-to. DNS status for the DNS section. Save calls PATCH /api/sending-domain/{id}/identity. Sender identity is per-domain, not account-global.
02a
DNS section · Managed variant
Managed-domain row 		The managed domain is always platform-managed. DNS is signed automatically — nothing for the tenant to configure here. Reads the explanatory copy. No actions. — No DNS state to surface; we own this.
02b
DNS section · Configured variant
Verified BYO 		BYO domain is verified and signing mail. Shows DKIM/Return-Path/DMARC summary, last-verified timestamp, and the "How to disconnect" explanation (the user removes the DKIM TXT record from their DNS provider — we don't have a one-click disconnect). Clicks Re-verify DNS or Reconfigure if needed. Reads the persisted DNS record metadata. Re-verify hits Postmark verify endpoints. Reconfigure launches the inline DNS wizard within this section (same Entri JWT-mint + showEntri flow as onboarding).
02c
DNS section · Disconnected variant
Recovery 		BYO domain was verified but a record (typically DKIM) has gone missing from DNS. Red banner with Reconnect and Forward fix to IT actions. Mail falls back to the managed domain in the background so sends keep going. Clicks Reconnect to re-launch Entri with the original record set, or Forward fix to IT to send a shareable link to their DNS admin. Specific record that's missing + intended value. Detection source. Either Entri's domain.record_missing webhook (Entri Monitor — paid, recommended) or our own reconciler job re-hitting Postmark verify on a 6h cycle. Reconnect re-launches entri.showEntri() with the original record set — no new Postmark domain created.
02d
DNS section · Verifying variant
Pending verification 		BYO domain has records in DNS but Postmark hasn't verified them yet. Per-record progress (DKIM / Return-Path / DMARC). Waits, or leaves the page (verification continues in background). Postmark verification status per record. Same polling as onboarding §02 row 03g.
02e
DNS section · Wizard variant (inline)
In-place reconfigure 		When the user clicks Reconnect (from Disconnected) or Reconfigure (from Configured), the DNS section swaps in place to the BYO wizard with its own mini sidebar. Sender identity above stays visible the whole time. Matches Voice's forwarding-rules pattern. Steps through Root → Subdomain → DMARC → Connect inside the DNS section. Same fields as the onboarding sub-journey. Same backend behavior as onboarding §02 row 03e.

Set as default
Row hover 		The default domain is used for all Engage broadcasts unless overridden on a specific automation or campaign. Star + "Default" pill in the row name cell signals the current default; a "Set as default" pill fades in on hover for non-default verified rows. Hovers a row, clicks "Set as default." Domain ID to set as default. PATCH /api/sending-domain/{id}/set-default. Toast confirms.

Add domain
Re-enters BYO sub-journey 		"+ Add domain" button at the top of the table. Enters the same 4-sub-step BYO sub-journey as onboarding (Root → Subdomain → DMARC → Connect). Identity seeded from the managed domain on entry. Steps through the sub-journey. On success, the new domain appears in the table (with the verifying state pill, animating to verified). Same as onboarding BYO sub-journey. Same as onboarding §02 rows 03b–03g.
05
Remove domain
Danger zone 		Only available for non-managed domains. Confirmation modal, then deletes the domain from our DB and the ESP. If the removed domain was the default, another domain auto-promotes to default. Clicks Remove in Danger zone → confirms in modal. Domain ID. DELETE /api/sending-domain/{id}. Backend calls Postmark DELETE /domains/{id} and deletes our row. (Disconnecting from DNS is a separate, manual process — see §02b explanation.)

Settings-specific rules

  • Managed domain is always present. Every Engage account has the managed domain as a row in the table, regardless of how many BYO domains they have. It can't be removed. When BYO domains disconnect, mail falls back to managed automatically.
  • Per-domain sender identity. Each domain owns its own prefix / displayName / replyTo. Editing the managed row's identity doesn't change BYO rows' identities and vice versa. The save bar in the detail page is scoped to the domain currently being edited.
  • Default badge ↔ Star pill. The Default pill in the row name cell is a single component: filled gold star + "Default" text for the current default, dashed-outline "Set as default" pill (hidden by default, shown on row hover) for non-default verified rows. Keyboard-accessible via focus-visible.
  • Disconnecting is manual. We don't have a one-click "disconnect DNS" button. The configured DNS card explains: to take a domain offline, remove the DKIM TXT record from your DNS provider. Once removed, our reconciler will catch it and the domain row will move to the Disconnected state.
  • Reconfigure vs Re-verify. Re-verify just re-polls Postmark — useful if the user just published records and wants to skip the polling backoff. Reconfigure re-launches the inline wizard from sub-step 1 (Root domain) so the user can change subdomain / DMARC / re-publish records.

Key design decisions · the "why"

Captured for future-Moshe (and future-Evan)

Managed-always, BYO-optional.

Klaviyo treats BYO as a choice up-front. We considered that and rejected it. SMBs hitting Engage for the first time aren't ready to think about DNS — but they also don't want to be blocked from sending. Auto-provisioning a managed domain gets them sending today; BYO becomes a clear opt-in upgrade for better deliverability rather than a setup hurdle. The two-card Email step makes the upgrade path visible without forcing it.

Voice-style DNS connection section.

The per-domain detail page splits into two sections: Sender identity (flat form) + DNS connection (stateful card with 5 variants — managed, configured, disconnected, verifying, inline wizard). Mirrors how the existing Voice/Assist product handles call forwarding setup. Gives the DNS state its own visible surface instead of burying it in a status pill, and means "Reconnect" has a natural home rather than living as a banner button. Voice-style consistency means engineers reading both products see the same pattern twice.

Sub-journey is a sidebar swap, not an inline wizard.

In the onboarding Email step, we briefly experimented with rendering the BYO wizard inline inside the DNS section. Too crowded — sender identity above + wizard sidebar + wizard content + wizard footer all competing in the same right pane. Replaced with a proper sub-journey: clicking Connect swaps the left sidebar entirely (parent button + sub-steps with circle markers), right pane gets the full canvas, footer Back/Continue map to sub-journey navigation. Inside settings, the DNS section's wizard variant works because it's already inside a master→detail descent — we're not stacking shells.

Per-domain sender identity.

Each domain owns its own prefix / displayName / replyTo — not a global Engage-account setting. Matches how Klaviyo, Postmark, and Customer.io think about multi-domain accounts. Lets a tenant send marketing from "Acme Insurance <marketing@send.acme.com>" and transactional from "Acme Insurance Tickets <noreply@notifications.acme.com>" with separate identities. We collect identity in the Subdomain sub-step (not at the end of the wizard, not silently inherited) so users see the BYO identity is its own thing from the start. Seeded from managed on first entry as a sensible default.

Reply-to deliverability nudge, not hard validation.

Best practice: reply-to address on the same root domain as the sending domain. We auto-suggest info@<sending-root> as the placeholder and show a soft warning chip if the user types a cross-root reply-to. We don't block — there are legitimate cases (third-party ticketing systems, IT-managed inbox routing) where a cross-root reply-to is correct. The warning chip is there to make the implicit deliverability risk explicit.

No one-click disconnect.

We don't expose a "Disconnect DNS" button. To disconnect, the tenant removes the DKIM TXT record from their DNS provider; our reconciler catches it and moves the domain to Disconnected state. Reasoning: disconnect is a real, in-DNS act that we don't have OAuth-write back into. If we shipped a button that just flipped a flag in our DB, mail would keep working (DKIM records would still be live) but our UI would lie about it. Better to instruct than to mislead.

"Continue while verifying" instead of a blocking spinner.

DNS propagation can take 1–5 minutes typically, up to 48h in pathological cases. Trapping the user on a spinner during onboarding kills the flow. The Verifying screen exists, shows per-record progress, but offers a "Continue while verifying →" button that advances to SMS while polling continues in the background. When verification lands, we notify them and the Email Channel's DNS section flips to Configured. Avoids the worst case where a tenant abandons setup because their DNS provider is slow.

Entri integration · technical contract

From the May 6 review call with Matt & Felipe

SDK install + setup

  • Plan: Startup ($249/mo). 600 domain connections/year covers current volume. We'll upgrade to Growth ($749/mo) when we approach the cap or add multi-domain-in-one-flow.
  • One Entri application per environment (dev / staging / prod). Per-partner branding deferred — the modal will show the platform mark, not partner marks, in v1.
  • checkDomain. Available on Growth plan only — Matt is checking with Joel about a one-off add-on. We'd call it on Subdomain sub-step blur to set expectations ("Cloudflare detected — automatic setup") but it's not gating. Skip in v1 if pricing doesn't work.

The Entri JS handoff

  • Server mints the JWT, never the browser. POST https://api.goentri.com/token with { applicationId, secret }. Returns { auth_token } valid 60 min. Backend wraps this as POST /api/sending-domain/entri-token.
  • dnsRecords array shape. We translate Postmark's response (DKIMPendingHost / DKIMPendingTextValue / ReturnPathDomain / ReturnPathDomainCNAMEValue) into Entri's { type, host, value, ttl } shape. host is relative to the root domain — strip the root suffix.
  • showEntri call. entri.showEntri({ applicationId, token, userId: "tenant_<id>", prefilledDomain, dnsRecords, onSuccess, onEntriClose }). userId is how we correlate webhooks back to the tenant.
  • DMARC. Send the DMARC record as part of the dnsRecords payload. Advanced DMARC validation/append is a premium-plan feature we don't need yet.

Webhook handling

  • Events we care about: domain.flow.completed (analytics), domain.added with propagation_status: "success" (fast-track Postmark verify), domain.propagation.timeout (mark Failed, notify admin), domain.record_missing + domain.record_restored (Entri Monitor — paid, used for disconnect detection).
  • Signature verification. SHA-256 over webhook_id + timestamp + client_secret (Entri's docs call this "HMAC" but the actual scheme is secret-suffix SHA-256 — match the code, not the prose). 5-min freshness window on Entri-Timestamp. Constant-time compare.
  • Source IP: 3.14.77.245 in production. Allow-list at the edge before webhook handler.
  • Idempotency. Webhooks retry on non-2xx. Store webhook_id and 200 on replay.

Postmark is the source of truth

  • Entri's domain.added · propagation_status: success means DNS records are live in public resolvers. That's necessary but not sufficient — Postmark has to independently verify DKIM before we'll DKIM-sign mail.
  • We poll PUT /domains/{id}/verifyDkim and PUT /domains/{id}/verifyReturnPath on backoff. Only when both return verified: true do we flip the domain to Verified state.
  • Same polling drives the Verifying state in both onboarding and settings.

Data model

Per-domain identity is the load-bearing shape

One sending_domain row per domain, scoped by business_id (multi-tenant). Each row owns its own sender identity.

sending_domain
├── id                  uuid
├── business_id         uuid          ── multi-tenant scope
├── full_domain         text          ── e.g. "send.acmeinsurance.com"
├── root_domain         text          ── e.g. "acmeinsurance.com"
├── subdomain           text          ── e.g. "send"
├── is_managed          bool          ── true for the platform-managed row
├── is_default          bool          ── exactly one per business is true
├── status              enum          ── pending | verified | disconnected | managed
├── purpose             text          ── informational: "Marketing" / "Transactional"
│
├── prefix              text          ── per-domain sender identity
├── display_name        text          ── ↳ each domain has its own "from" line
├── reply_to            text          ── ↳ scoped per-domain, not account-wide
│
├── dmarc_enabled       bool          ── whether DMARC was included in BYO setup
├── postmark_domain_id  bigint        ── FK into Postmark's domains API
├── last_verified_at    timestamptz
└── created_at          timestamptz

Invariants

  • Exactly one row per business has is_managed = true. It's provisioned at tenant-create time, never deleted. If all BYO domains disconnect, mail falls back to this row.
  • Exactly one row per business has is_default = true. Removing the default auto-promotes another verified non-managed row (or the managed row if no BYO exists).
  • The managed row's identity is editable. Even though we own the DNS for the managed domain, the tenant still controls prefix / display name / reply-to.
  • Per-row identity, no inheritance at runtime. Adding a new BYO domain seeds identity from the managed row at create time as a UX default — but once saved, the new row's identity is independent. Editing the managed row later doesn't change BYO rows.
